## 1. What this project is

PaddockME is a two-sided agistment marketplace: **livestock owners** post **requests** for grazing; **landowners** list **paddocks**; they form an **agreement** (shared "workspace"); **transport providers** move stock.

- Stack: **Next.js (App Router) + TypeScript + Tailwind**, **Supabase** (Postgres + Auth + RLS), deployed on **Vercel** (project `paddockme-oz51`, prod domain `paddockme-oz51.vercel.app`).
- Repo: `github.com/jwcoleman87-collab/paddockme`, working branch `main`. Local checkout: `C:\Users\James Coleman\Desktop\paddockme`.
- Build/verify locally: `npm run typecheck` (= `tsc --noEmit`) and `npm run build`. The Linux sandbox can't run the Windows‑compiled SWC build binary, so typecheck is the main local gate; the real `next build` runs on Vercel on every push to `main`.

## 2. The single most important architectural fact

There are **two parallel data worlds**, and most bugs this session came from crossing them:

1. **Prototype / demo world** — `src/lib/dummyData.ts` (seed personas Dale/Brett/Wayne = ids `farmer-a`/`farmer-b`/`driver-1`, seed requests/agreements/transport) + `src/lib/prototypeStore.ts` (localStorage). Used for unauthenticated demo visitors and the investor walkthrough.
2. **Real world** — Supabase tables (`paddocks`, `agistment_requests`, `matches`, `agreements`, `agreement_sections`, `messages`, `profiles`). Used for signed‑in accounts.

Key rules:
- `src/lib/data/repositories.ts` is a **`"use client"` module** (browser Supabase client + localStorage). **Server Components must NOT import it** — doing so throws `Attempted to call X from the server`. That exact bug 500'd `/listings` (commit `50a10ff`).
- For **server-side** Supabase reads, use **`src/lib/data/serverPaddocks.ts`** (cookie-aware server client from `@/lib/supabase/server`). This is the pattern to copy for any new server-rendered real data.
- Real-account gating pattern: a server page calls `getCurrentUserProfile()` (`@/lib/supabase/currentUser`, server-only). If truthy → render real data / `RealAccountEmptyState`; else → render seed demo data.
- **Mock paddock listings were deliberately emptied** (`paddockListings = []` in dummyData) per the owner's request — demo paddock surfaces are intentionally empty now. Personas/requests/transport seed arrays still exist (used only on unauthenticated paths).

## 6. Practical / environment gotchas (read before editing)

- **No GitHub credentials in the agent sandbox.** Cannot `git push` from the sandbox. Commits are made in the mounted repo, then pushed from the **Windows machine** (this session used **Git GUI**: Open Recent → decline the gc prompt → Push → main → origin). Vercel auto-deploys `main`.
- **Windows mount flips line endings (LF↔CRLF)**, which makes `git status` show ~60–90 files "changed" that are pure line-ending noise. **Only stage the specific files you edited** (`git add -- <paths>`), never `git add -A`. Recommend adding a `.gitattributes` with `* text=auto eol=lf` to stop this (not yet done; time it so it doesn't collide with parallel branches).
- **File-write truncation:** large writes via the editor tool intermittently truncated mid-file on this mount. Writing files via a Python heredoc (`python3 - <<'PY' ... open(p,"w").write(...)`) was reliable. Always `tsc --noEmit` after edits to catch a truncated tail.
- There are parallel branches (`claude/*`, `codex/*`) and **uncommitted real edits** in the working tree from other sessions — don't sweep them into commits.
- Verify deploys via the Vercel API/MCP (`list_deployments` / `get_deployment`, team `team_dCW82GqfRmq0Pl71eMgYL59J`, project `prj_iX2WcfTZqUICl5UVgbnsB8JEsfQo`). Runtime error logs there were essential for diagnosis.
