## SEV-1 Critical

### BUG-01 — Google Maps API key committed to source control

- **Location:** `src/components/PaddockMap.tsx:125`, `src/app/(app)/transport/available/CapacityClient.tsx:33`, `src/app/(app)/transport/TransportJobsClient.tsx:72`
- **What:** The key `AIzaSyAG3EVoUUNfk0amP7J40Dy1NpmGG3_1L18` is hardcoded as a fallback in all three files. It is now in git history permanently. A comment says it is HTTP-referrer-restricted, but that restriction can be bypassed (e.g. via curl with a spoofed Referer header, or if the restriction was ever misconfigured). More importantly: once in git history, the key can never be rotated out of history without a full `git filter-repo` rewrite.
- **Expected:** API key in `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` env var only. If the var is absent, the map should degrade gracefully (show the `DummyMap` fallback) rather than falling back to a hardcoded credential.
- **Repro:** `git log -S 'AIzaSyAG3EVoUUNfk0amP7J40Dy1NpmGG3_1L18' --all`
- **Fix:** Remove the `?? "AIzaSy..."` fallback from all three files. Verify the env var is set in Vercel. Add `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` validation in `src/lib/supabase/env.ts` (or a new `env.ts` for Google) so the map component receives `null` and renders the static fallback when the key is absent.

## SEV-2 High

### ~~BUG-02~~ — RETRACTED: `proxy.ts` is correct for Next.js 16

- **Status:** False positive — retracted 2026-06-08.
- **What happened:** The scan misidentified `src/proxy.ts` as a misnamed middleware file. Next.js 16.2.6 renamed the middleware file convention from `middleware.ts` to `proxy.ts`. The original file is correct and IS being picked up by the framework — confirmed by `ƒ Proxy (Middleware)` in the production build output. A rename to `middleware.ts` actually introduces a deprecation warning (`"The 'middleware' file convention is deprecated. Please use 'proxy' instead"`). No action needed.

### BUG-03 — Stripe checkout session creation has no error boundary

- **Location:** `src/app/api/payments/transport/checkout/route.ts:70`
- **What:** `stripe.checkout.sessions.create(...)` is called with `await` but is not wrapped in try/catch. If Stripe throws (expired test key, network failure, API rate limit, invalid `currency` enum), the route crashes with an unhandled exception and returns an HTML 500 page instead of a structured JSON error. The frontend payment button then fails silently (JSON parse error on the HTML response).
- **Expected:** Stripe call wrapped in try/catch returning `{ error: message }` with an appropriate HTTP status (502 or 503).
- **Repro:** Set `STRIPE_SECRET_KEY` to an invalid value and hit the checkout endpoint — you get an HTML 500.
- **Fix:** Wrap lines 70–91 in `try { ... } catch (err) { return NextResponse.json({ error: err instanceof Error ? err.message : "Stripe error" }, { status: 502 }); }`.

### BUG-04 — Stripe webhook handler has no error boundary around the ledger write

- **Location:** `src/app/api/webhooks/stripe/route.ts:41`
- **What:** `await recordStripeCheckoutCompleted(session, event.id)` is called without try/catch. If the Supabase write fails, the webhook returns 500, Stripe retries the event indefinitely (up to 72 h), and the eventual successful retry inserts a duplicate `payment_events` row (the idempotency guard is `provider_event_id`, so duplicates are blocked, but repeated 500s generate noise and alert fatigue).
- **Expected:** The ledger write wrapped in try/catch; on failure, log the error and return 200 so Stripe stops retrying, or return 500 only if retry is safe.
- **Fix:** Wrap lines 38–43 in try/catch; on catch, log the error and decide whether to 200 (idempotent-safe) or keep 500.

### BUG-05 — `getAgreement()` and `getTransportJob()` silently return seed record on invalid ID

- **Location:** `src/lib/dummyData.ts:1356`, `src/lib/dummyData.ts:1360`
- **What:** Both functions use `?? agreements[0]` / `?? transportJobs[0]` as a fallback. Navigating to `/workspace/bad-id` renders Agreement 1's full workspace. Navigating to `/transport/bad-id` renders Transport Job 1's room. Deep links from old data, expired sharing links, or typos all silently show wrong data rather than a 404.
- **Expected:** Return `undefined` on no-match; callers (`WorkspaceRouteClient`, `TransportRouteClient`, `TransportDetailPage`) show a "not found" card.
- **Fix:** Remove the `?? [array][0]` fallbacks. Both callers already have a `if (!job)` / `if (!agreement)` guard that renders an empty-state card — they just never trigger because the fallback always returns something.

### BUG-06 — `openWorkspace()` in ListingDetailClient has no error handler

- **Location:** `src/app/(app)/listings/[id]/ListingDetailClient.tsx:65`
- **What:** `async function openWorkspace()` calls `await openAgreementWorkspace(listing.id)` with no try/catch. If the function throws (Supabase error, network failure), the rejection is unhandled. React does not catch async event-handler rejections — the button silently does nothing and the error disappears.
- **Expected:** try/catch around the `await`, showing a flash error message on failure.
- **Fix:** Wrap the body of `openWorkspace` in try/catch and call `flash("Could not open workspace. Please try again.", "error")` in the catch.

### BUG-07 — No `error.tsx` in any route group (zero error boundaries)

- **Location:** `src/app/(app)/`, `src/app/(auth)/`, `src/app/` — all missing `error.tsx`
- **What:** If any server component throws (Supabase misconfiguration, unexpected null, etc.), Next.js shows a full-page white error screen with no navigation or recovery path. The user is stuck.
- **Expected:** At minimum `src/app/(app)/error.tsx` with a "Something went wrong" message and a "Go home" link.
- **Fix:** Add `src/app/(app)/error.tsx` and `src/app/error.tsx` as client components with `"use client"` and the `({ error, reset })` signature. One card with a reset button is enough.

### BUG-08 — No `not-found.tsx` in any route group

- **Location:** `src/app/(app)/`, `src/app/` — both missing `not-found.tsx`
- **What:** Because `getAgreement()` and `getTransportJob()` fall back to seed data (BUG-05), a 404 UI never shows. Even after fixing BUG-05, without `not-found.tsx` Next.js will show the default Next.js 404 page with no site branding or navigation.
- **Expected:** Branded not-found page with links back to `/agreements` and `/listings`.
- **Fix:** Add `src/app/not-found.tsx` (global) and optionally `src/app/(app)/not-found.tsx`.

## SEV-3 Medium

### BUG-09 — Deprecated Google Maps APIs generate console warnings on every map visit

- **Location:** `src/components/PaddockMap.tsx`, `src/app/(app)/transport/TransportJobsClient.tsx` — confirmed by live scan on `/map` and `/map?mode=driver`
- **What:** Three deprecated APIs are called on every render: `google.maps.DirectionsService` (deprecated Feb 2026), `google.maps.Marker` (deprecated Feb 2024), `google.maps.DirectionsRenderer` (deprecated Feb 2026). These generate repeated console warnings. Google hasn't announced a shutdown date yet, but the APIs could be sunset without a long warning window.
- **Expected:** Migrate to `google.maps.routes.RoutesClient` and `google.maps.marker.AdvancedMarkerElement`.
- **Fix:** Plan a migration sprint. In the short term, suppress the console noise by upgrading to the current `@googlemaps/js-api-loader` version if it provides newer defaults.

### BUG-10 — Debug placeholder strings exposed to real Supabase auth users

- **Location:** `src/lib/data/repositories.ts:70–71`
- **What:** `mapProfileRow()` sets `tagline: "Supabase profile"` and `bio: "Profile loaded from Supabase."` on every row returned from the real Supabase `profiles` table. These strings render in the Profile page for any signed-in user.
- **Expected:** Empty string, `null`, or profile data from the actual row.
- **Fix:** Set `tagline: row.tagline ?? ""` and `bio: row.bio ?? ""`, or remove the fields from `mapProfileRow` entirely and let the UI handle empty state.

### BUG-11 — Async data refreshes silently drop errors in three client components

- **Location:** `src/app/(app)/agreements/AgreementsClient.tsx:83`, `src/app/(app)/workspace/[id]/WorkspaceRouteClient.tsx:27`, `src/app/(app)/listings/[id]/ListingDetailClient.tsx:34`
- **What:** All three use `void Promise.all([...]).then(...)` without a `.catch()`. If the repository calls reject (Supabase down, network offline), the UI stays with stale server-rendered data and no error indication. The user sees no error, no retry option, and assumes the page is just slow.
- **Expected:** `.catch(err => { /* show error state or flash */ })` appended to each Promise chain.
- **Fix:** Add `.catch(() => { /* at minimum a flash or a local error state boolean */ })` to each `void Promise.all(...)` call.

### BUG-12 — No `loading.tsx` for any async server page

- **Location:** All pages in `src/app/(app)/` and `src/app/(auth)/`
- **What:** Async server pages (`profile/page.tsx`, `listings/[id]/page.tsx`, `matches/page.tsx`, etc.) show a blank white screen while the server renders. No skeleton or spinner is rendered. On slow connections this looks like a broken page.
- **Expected:** At minimum `src/app/(app)/loading.tsx` with a simple `<div className="animate-pulse">` skeleton.
- **Fix:** Add a route-group-level `loading.tsx` (one file covers all routes in the group). A simple full-height pulse skeleton is enough.

## SEV-4 Low

### BUG-13 — `mapMessageRow` typed as `row: any`

- **Location:** `src/lib/data/repositories.ts:787`
- **What:** The function signature `function mapMessageRow(row: any, threadId: string)` means TypeScript cannot catch field-access errors. A column rename in the Supabase `messages` schema won't produce a compile error.
- **Fix:** Type `row` as `Tables<"messages"> & { profiles?: { full_name: string | null } | null }` (Supabase's join shape).

### BUG-14 — `skipLibCheck: true` in tsconfig

- **Location:** `tsconfig.json:6`
- **What:** Suppresses type errors originating in `.d.ts` library files. Safe for most projects but can mask issues when libraries update their types.
- **Fix:** Low priority. Note for future: set `skipLibCheck: false` once all library type errors are resolved.

### BUG-15 — Transport earnings page renders "placeholder" and "out of scope" copy in production

- **Location:** `src/app/(app)/transport/earnings/page.tsx:12`
- **What:** The `PageHeader` `description` prop contains "A lightweight placeholder for driver earnings. Payments and payout logic are intentionally out of scope for this MVP build." This text is visible to any real user navigating to `/transport/earnings`. The `description` field currently says "Coming later" in the card body, which is cleaner.
- **Fix:** Remove the `description` prop from `PageHeader` on this page, or replace with "Earnings history coming soon."

### BUG-16 — `Avatar.tsx` renders profile images with `alt=""`

- **Location:** `src/components/Avatar.tsx:65`
- **What:** `<Image src={src} alt="" ...>` — empty alt text marks the image as decorative. An avatar that represents a specific person (Dale, Brett, Wayne) is not decorative. Screen readers skip it entirely.
- **Fix:** Pass `alt={name}` (the user's display name) into the `Avatar` component and thread it through to the `Image` tag.

### BUG-17 — Google Maps fallback API key duplicated in three separate files

- **Location:** `src/components/PaddockMap.tsx:123–125`, `src/app/(app)/transport/available/CapacityClient.tsx:31–33`, `src/app/(app)/transport/TransportJobsClient.tsx:70–72`
- **What:** The same fallback expression is copy-pasted in three places. After fixing BUG-01, all three will need updating independently. One missed file means the key persists.
- **Fix:** After BUG-01 is resolved (fallback removed), extract the env-var read into a single shared `src/lib/googleMapsKey.ts` module and import it.

### BUG-18 — `dummyData.ts` "Verification placeholder" strings may surface in profile UI

- **Location:** `src/lib/dummyData.ts:476`, `src/lib/dummyData.ts:543`
- **What:** Both demo farmer profiles have `{ label: "ABN", status: "Pending", detail: "Verification placeholder" }` in their `verifications` array. If this array is rendered anywhere in the Profile UI, users see the word "placeholder".
- **Fix:** Replace "Verification placeholder" with a realistic value like `"Pending ABN verification"`.

## What Was Not Scanned

- **Auth-gated routes that require real sign-in**: The `bug-scan@example.com` account could not sign in (account doesn't exist; sign-up requires email confirmation which can't be automated without inbox access). App routes were scanned using the demo-persona localStorage pattern, which bypasses Supabase auth entirely.
- **POST/mutation flows**: The scan only checked page loads via GET. Forms (new request, new listing, send message, post capacity, pay for transport) were not tested.
- **Supabase RLS correctness at runtime**: Policies were read from migration files. Whether the applied Supabase instance matches the migrations was not verified.
- **Stripe webhook delivery**: The `/api/webhooks/stripe` endpoint was not triggered during the scan.
- **Mobile-specific layout issues**: Viewport was switched to 375×667 on each page but visual layout was not inspected — only that the page rendered without a JS crash.
- **Cross-browser compatibility**: Only Chromium was used.
- **Performance profiling**: TTI was measured but no Lighthouse or Web Vitals audit was run.
- **`/transport/calendar` route**: Not in the scan list — this route exists (`transport/calendar/page.tsx`) but was not covered.
